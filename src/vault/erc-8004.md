ERC-8004 is an Ethereum standard that adds three on-chain registries for AI agent identity, reputation, and validation — providing censorship-resistant, composable agent trust infrastructure that works across organizational boundaries without prior coordination.

## The Problem It Solves

A2A and MCP solve agent discovery and communication within trust boundaries. When agents cross organizational boundaries, DNS and TLS are not enough: there is no standard way to discover, verify, or audit an agent's track record without a prior bilateral relationship. ERC-8004 provides a public, tamper-resistant layer that any agent or service can query.

## Three Registries

**Identity Registry.** Each agent receives an NFT (ERC-721) linking to flexible endpoints: A2A agent cards, MCP servers, ENS names, DIDs, wallets on any chain. The NFT is the global identifier — portable across chains and platforms, not owned by any single provider.

**Reputation Registry.** Signed feedback with contextual tags rather than a single aggregate score. Past callers rate agents with structured ratings ("accurate," "fast," "reliable") that future callers filter by what matters to their use case. Payment receipts (via [[x402]]) prove the reviewer actually used the service — Sybil resistance built into the feedback mechanism.

**Validation Registry.** For high-stakes outputs, agents can request independent verification from validators. Supported methods: stake-secured (EigenLayer), zero-knowledge ML proofs, trusted execution environments (Phala, Near.AI), and trusted judges. Validators respond on-chain with a score and evidence hash. The requesting agent chooses the validation method; the record cannot be deleted.

## The Trust Flow

1. Client agent looks up service agent identity in the Identity Registry
2. Checks reputation in the Reputation Registry
3. Calls the service, pays with [[x402]]
4. Optionally requests output validation
5. Submits feedback — backed by on-chain payment receipt as proof of use

Each step produces an immutable on-chain record. The full chain is auditable by any party.

## Deployment State

Went live on Ethereum mainnet January 29, 2026. Co-developed by Ethereum Foundation, Consensys, Google, and Coinbase. Deployed across 18+ EVM-compatible chains (Polygon, BNB Chain, Base, Arbitrum, Mantle, Avalanche, others) using singleton contracts — all agents share the same registry per chain.

## Honest Limitations

- Sybil attacks remain possible (fake agents inflating reputation before consequences arrive)
- Capability claims are not verified — advertised capabilities may not be functional
- On-chain settlement creates audit trails that cannot be deleted, but does not prevent initial fraud

The combination of reputation (community-sourced, payment-backed) and validation (independent, stake-secured) provides layered trust signals, but is not a guarantee of agent reliability.

## On-Chain vs. Off-Chain

ERC-8004 on-chain: censorship resistance, composability with DeFi primitives, cross-organizational use.
A2A agent cards, MCP servers off-chain: lower latency, simpler integration, internal operations.

Most organizations will use both: on-chain for cross-organizational trust where no prior relationship exists, off-chain for internal agent communication.
