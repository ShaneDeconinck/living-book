## 1. IETF OAuth WG — Agent Authorization Convergence Moment

**Status:** Four competing agent auth proposals are now active simultaneously in the OAuth WG:

- **draft-klrc-aiagent-auth-00** (Kasselman et al., March 2) — AI Agent Authentication and Authorization framework
- **draft-barney-caam-00 / CAAM** (Jonathan Barney, February 24) — Contextual Agent Authorization Mesh, token delegation
- **AgentID** (Warren Parad et al., March 15) — identity protocol for autonomous AI agents within OAuth. Note: filed as Internet-Draft but not confirmed in datatracker as of yesterday's scout; status unchanged.
- **A2A OAuth Transaction Token Profile** (George, March 16) — OAuth profile for A2A communication

**The signal:** On March 16, a thread titled "Overlap of AI related proposals" appeared with multiple heavy hitters posting — Dick Hardt, Bjorn Hjelm, Alex Babeanu. The WG is now explicitly trying to rationalize these competing proposals.

**Relevance:** Very high. Fragmentation either resolves into a converged standard or splinters into competing ecosystems. The outcome determines what the book's OAuth-for-agents chapter says about the authoritative path.

**Actionable:** Monitor. The "Overlap" consolidation discussion is what matters. A WG liaison document or consensus call is the milestone.

## 2. MCP Roadmap — Enterprise Security Gaps Named

**Status:** MCP spec version is still 2025-11-25. No new spec release in 2026. The roadmap page (updated March 5) is the most substantive new content.

Two security SEPs in progress:
- **SEP-1932 (DPoP)** — proof-of-possession for MCP tokens
- **SEP-1933 (Workload Identity Federation)** — enterprise SSO away from static secrets

Priority enterprise gaps explicitly named in the roadmap: audit trails/observability, enterprise-managed auth (SSO away from static secrets), gateway/proxy authorization propagation, configuration portability across MCP clients. (Conformance test suites are a separate validation priority, not an enterprise readiness gap.)

**Relevance:** High. The named gaps map directly to book chapters: agent-observability (audit trails), agent-communication (gateway auth propagation), agent-identity (workload identity federation). DPoP is the concrete mechanism the book should cite when discussing token binding for MCP.

**Actionable:** The book can now name DPoP (SEP-1932) and Workload Identity Federation (SEP-1933) as the specific mechanisms MCP is implementing for enterprise auth hardening.
